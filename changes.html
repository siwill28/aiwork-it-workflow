Design Principles Confirmed This Audit
1
Actions are atomic and context-independent. Skills carry and reason with context across multiple steps. When building new functionality: if the task requires no surrounding context, build it as an action. If it needs to reason across state or prior steps, build it as a skill.
2
Assignment always happens at the workstream level before skills are called. Confirmed for both IT-USA Tasks (Task Type Verifier at Step 4) and New Incidents - US (Assign Incident To Yourself Action). No skill in either workstream should self-assign.
3
Early workstream-level filters handle known ticket patterns before deeper processing. SAP migration, wrong ticket type, self-generated tickets, FedRAMP — all caught at the workstream level before any skill executes.
4
Comment first, then state change. On Hold is set after the agent posts a comment to the user — never before. The agent must communicate before changing ticket state at any point in either workstream.
5
Try to resolve fast, ask only when needed. KB search runs before clarifying questions. If KB produces a confident resolution path, skip Gather Additional Information entirely and go directly to Provide Resolution. This minimises user wait time.
6
Task Type Verifier is the assignment gate for IT-USA Tasks. No skill in that workstream self-assigns. The gate is at Step 4 — all Step 6 routing skills inherit an already-assigned ticket.
7
Approval satisfied by submitter identity. If opened_by matches a named required approver in the access policy, the ticket submission itself constitutes approval. The agent proceeds directly to granting access without requesting separate approval.
8
Skills are orchestrators, not communicators. Application Permission Requests directs the flow. Communication is owned by the skills and actions that perform the actual work — not by the orchestrating skill. Avoids duplicate notifications.
