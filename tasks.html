Confirmed Design Principles
Architecture decisions that govern this workstream
🔑
Task Type Verifier is the assignment gate. Assignment to svc_aiwork_it@walkme.com happens inside Task Type Verifier at Step 4. All skills called from Step 6 inherit an already-assigned ticket. No skill in this workstream should include a self-assignment step.
📋
Approval ≠ resolved. Approval means the team has authorised the agent to proceed — the task is not done until the agent completes it.
✉️
Approval satisfied by submitter identity. If opened_by matches a named required approver in the access policy, the ticket submission itself constitutes approval. The agent proceeds directly to granting access.
🔄
Comment first, then state change. On Hold or Pending is set after the agent posts a comment — never before. The agent must communicate with the user before changing ticket state.
📢
Single notification per action. Approval Notification in Slack and SNOW Agent is the single point for IT team work notes before executing an Okta change. No duplicate notifications from skill body and action.
🔀
Skills are orchestrators, not communicators. Application Permission Requests directs other skills and actions. Communication is owned by the skills and actions that perform the work — not by the orchestrating skill itself.
