Operating Guidelines
Scope, eligibility, and operating principles
Process when
✓Incident status is New or In Progress AND unassigned
✓Incident is already assigned to AI Work
✓Incident was reassigned from another agent to AI Work
Stop immediately if
✗Incident assigned to any other team or user
✗Incident is already resolved
✗Incident opened by ai.work indicating emergency IT channel
✗Incident is related to FedRAMP (out of scope)
Operating principles
📋First action is always ServiceNow | Get Incident Details including full comment thread
✓Resolution must be explicitly confirmed by the user before closing (unless 3-day inactivity applies)
⏸On Hold / Awaiting Caller is set after agent posts a comment — only when waiting for user response to a resolution or clarifying questions
📊Use ServiceNow | Get Table Schema if a field value is missing before updating
Tone
💬Write directly to the user, not about them. Polite and professional, not robotic. No internal IT jargon. Acknowledge frustration.
⚡Skip asking questions if KB has a clear single path, if system checks identified a root cause, or if you can already resolve with what you know.
🏛
Confirmed Design Principles
Architecture decisions governing this workstream
📖
Read full history before acting. Collect Ticket Information fetches the complete comment thread. The agent must understand the current state of the conversation and jump to the appropriate step rather than always running the full pipeline from the top.
⚡
Try to resolve fast — ask only when needed. Troubleshooting runs before Gather Additional Information. If KB produces a confident resolution path, skip steps 4, 5, and 6 entirely and go directly to Provide Resolution. Only ask clarifying questions when KB is genuinely inconclusive.
💬
Comment first, then state change. On Hold / Awaiting Caller is set after the agent posts a comment to the user — never before. The agent must communicate before changing ticket state.
🔄
Monitor means re-read, acknowledge, advance. After each user reply, the agent must re-fetch the full ticket and comment thread, review what has already been provided, acknowledge what the user said, and advance to the next logical step rather than repeating prior steps.
⏱
4-exchange escalation limit. After 4 exchanges (one AI comment + one user reply = one exchange), escalate by changing assignee to unknown and adding a comment explaining why. This prevents troubleshooting loops when KB quality is insufficient.
📷
Agent cannot view attachments. If a user attaches a screenshot of an error, Generate Clarifying Questions explicitly asks the user to paste the exact error text as a comment so the agent can read it.

Open Items
Gaps, monitoring items, and pending decisions
⚠
Troubleshooting exit condition needs updating. The active instruction says "if a clear resolution path was found, proceed directly to Provide Resolution" but does not say to skip Application Access Verification, System Data Collection, and Gather Additional Information. Without this explicit stop, the agent falls through to those steps and generates a second comment. Add: "skip steps 4, 5, 6 and proceed directly to Provide Resolution then Monitor and Follow Up."
⚠
General Troubleshooting — redundant assignment check. The skill opens with a check to make sure the ticket is assigned. Assignment is already handled at the workstream level (Step 1 Assign). The check is harmless but redundant. Low priority cleanup.
ℹ
Monitor — Generate Clarifying Questions example questions commented out. All example questions in Core Clarifying Question Categories are commented out as a deliberate test. Evaluating whether the agent generates contextually appropriate questions from category headers alone. If question quality drops, restore examples as guidance. If quality holds or improves, remove commented lines permanently.

Wrong ticket number in approval UI is platform behavior. When a tool requires approval (addTaskComment_variant_2), the approval UI shows INC0010001 rather than the actual ticket number. This is a platform artifact from how ai.work processes tool calls requiring human approval. The comment posts to the correct ticket once approved. Not a bug.
